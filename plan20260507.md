# 重构计划：移除 WI 与并行执行上下文，建立新权威模型

## Context

WI 和并行执行上下文（execution_group / execution_branch / feature_branch）是为多 agent 多分支并行开发设计的。用户的实际工作模式是 **单分支顺序推进**（一个变更开一个分支），这两套机制的收益均为零，但引入了：

- design.md ↔ work-items/*.yaml 双源一致性负担
- 额外的 CLI 命令（add-work-item、set-execution-context 等）
- meta.yaml 中 7 个仅服务于并行模型的字段
- 大量只服务于并行/WI 的 gate 校验逻辑

**目标**：移除 WI 和并行执行上下文，将 scope 和追溯职责回归 design.md 已有字段，建立简洁的单一权威模型。

---

## 一、新权威模型

删除 WI 和并行上下文后，各文档的管辖边界如下：

| 文档 | 管辖领域 | 关键字段 |
|---|---|---|
| **meta.yaml** | 阶段状态、实现基线、authority repair | phase, status, implementation_base_revision, active_authority_repair, gate_evidence |
| **design.md §4** | scope 边界（HTML 标记包围的 glob 列表） | `可修改` / `不可修改` |
| **design.md §5** | 契约引用 | contract_ref, summary |
| **design.md §7** | 实现切片与追溯 | slice_id, goal, requirement_refs, acceptance_refs, verification_refs, test_case_refs |
| **design.md §8** | 实现阶段输入 | runbook, contract, view, verification |
| **testing.md** | TC/RUN/HANDOFF、完成等级 | TC→ACC 追溯，RUN 结果与 completion_level，HANDOFF 阶段交接 |
| **contracts/*.md** | 契约冻结状态 | status (draft/frozen)，不维护 consumers |

**不再出现于任何文档的字段**：
focus_work_item, active_work_items, wi_id, dependency_refs, dependency_type, execution_group, execution_branch, feature_branch, branch_execution

**注意**：allowed_paths / forbidden_paths 作为**字段名**从 WI yaml 和 design.md 中消失，但 authority repair record（`authority-repairs/*.yaml`）的 `allowed_paths` 字段**保留**，这是受控越界机制的一部分，与 WI scope 无关。

### 不可变规则一：scope 权限矩阵

所有 gate（scope、phase-capability、dirty-worktree）共享同一个权限判定 helper `file_permission(file, phase)`。

**文件分类**：

| 文件类别 | 文件模式 | 规则 |
|---|---|---|
| hardcoded-forbidden | `versions/**` | 所有阶段拒绝修改 |
| authoritative-doc | `spec.md`, `design.md` | 所有阶段拒绝修改（除非 authority repair） |
| deployment-doc | `deployment.md` | 只在 Deployment 阶段可修改 |
| evidence-doc | `meta.yaml`, `testing.md` | Implementation/Testing/Deployment 阶段自动放行 |
| agent-doc | `CLAUDE.md`, `AGENTS.md` | 所有阶段自动放行 |
| repair-doc | `authority-repairs/*.yaml` | 有活跃 repair 时放行 |
| review-doc | `reviews/**` | phase transition 时放行 |
| contract-doc | `contracts/*.md` | 只在 authority repair 时可修改；frozen contract 任何情况下不可修改 |
| design-scope-allowed | design.md §4 可修改 glob | Implementation 阶段允许修改（匹配 glob 的文件） |
| design-scope-forbidden | design.md §4 不可修改 glob | 所有阶段拒绝修改 |
| src-code | 其他文件 | Implementation 阶段按 design-scope 判定；其他阶段拒绝 |

**判定优先级**（短路求值）：

```
1. authority-repair 生效中？→ 按 repair allowed_paths 判定
2. 文件匹配 hardcoded-forbidden？→ 拒绝
3. 文件匹配 authoritative-doc？→ 拒绝
4. 文件匹配 design-scope-forbidden？→ 拒绝
5. 文件匹配 evidence-doc 且当前阶段允许？→ 放行
6. 文件匹配 agent-doc？→ 放行
7. 文件匹配 deployment-doc 且 phase=Deployment？→ 放行
8. 文件匹配 design-scope-allowed 且 phase=Implementation？→ 放行
9. 其他 → 拒绝
```

**共享 helper 签名**：
```bash
is_lifecycle_exempt(file, phase)  → 0=exempt, 1=not exempt
design_scope_allowed(file)        → 0=allowed, 1=not allowed
design_scope_forbidden(file)      → 0=forbidden, 1=not forbidden
file_permission(file, phase)      → 0=allow, 1=deny
```

gate_scope()、gate_phase_capability()、check_dirty_worktree() 统一调用 `file_permission()`，不再各写一套。

### 不可变规则二：slice 完成度算法

**数据源**：design.md §7 每个 slice 的 `test_case_refs` → testing.md 中对应 TC 的 latest RUN。

**规则**：
- TC 只被一个 slice 引用时，RUN 可省略 slice_ref，自动归属该 slice
- 同一 TC 被多个 slice 引用时，RUN 必须写 slice_ref
- slice 完成等级 = 其 test_case_refs 所有 TC 的 latest RUN 中 **最低** completion_level
  - 例如：SLICE-001 有 TC-1 (integrated_runtime) 和 TC-2 (api_connected)，则 SLICE-001 完成等级 = api_connected
- slice 未完成条件：任一 TC 的 latest RUN 不是 pass，或 completion_level 低于 required_completion_level

### 不可变规则五：迁移安全边界

- **install-workspace.sh 不做任何项目数据操作**（不删 work-items/、不改 meta.yaml）
- 迁移只通过 `codespec migrate-remove-wi <project_root>` 显式执行

### 追溯链

**之前**：`REQ → ACC → VO → TC → WI → RUN`
**之后**：`REQ → ACC → VO → TC → RUN`

### SLICE-* 规则

- 格式：`SLICE-NNN`（三位数字，如 SLICE-001）
- 唯一性：design.md §7 内 slice_id 不可重复（gate 校验）
- HANDOFF 必填 `slice_refs: [SLICE-001]`
- RUN 的 `slice_ref`：TC 只被一个 slice 引用时可省略，被多个 slice 引用时必填
- 不参与状态管理：slice_id 不出现在 meta.yaml 中

---

### contract 新模型

- contracts/*.md 只维护 status (draft/frozen) + frozen 元数据
- **删除 consumers 字段**
- gate 校验规则：
  1. design.md §5 引用的 contract 文件必须存在
  2. frozen contract 未被修改
  3. **进入 Implementation 前**（implementation-start gate），design.md §5 中引用的 contract 必须 status: frozen 且 freeze 元数据完整（保留现有冻结门槛）
- 删除 consumers 不等于删除 freeze maturity

### 2.2 §4 系统结构 — 实现边界（替换 impacted/unchanged_surfaces）

用 HTML 注释标记限定机器可解析区间（与现有 `<!-- CODESPEC:DESIGN:READING -->` 风格一致），用 awk 在标记区间内提取 glob。

```markdown
<!-- CODESPEC:DESIGN:STRUCTURE -->
## 4. 系统结构

- system_context: [当前系统入口、调用链或模块背景]
- data_flow: [关键数据流或状态流]
- external_interactions:
  - name: [外部系统或 none]
    direction: inbound/outbound/both
    protocol: [HTTP/event/SDK/file/none]
    failure_handling: [失败、超时、重试或降级策略]

<!-- CODESPEC:SCOPE_ALLOWED -->
### 可修改路径

- `src/auth/**` — 认证模块重构
- `src/middleware/token.ts` — Token 中间件
<!-- CODESPEC:SCOPE_ALLOWED_END -->

<!-- CODESPEC:SCOPE_FORBIDDEN -->
### 不可修改路径

- `src/billing/**` — 不在本次变更范围
- `contracts/api-auth.md` — 已冻结契约
<!-- CODESPEC:SCOPE_FORBIDDEN_END -->
```

gate 解析逻辑：

```bash
# 提取可修改 glob（awk 限定在标记区间内，不依赖 grep -P）
awk '/<!-- CODESPEC:SCOPE_ALLOWED -->/{f=1;next} /<!-- CODESPEC:SCOPE_ALLOWED_END -->/{f=0} f && /^- `/{gsub(/^`|`$/,"",$2); print $2}' design.md

# 提取不可修改 glob
awk '/<!-- CODESPEC:SCOPE_FORBIDDEN -->/{f=1;next} /<!-- CODESPEC:SCOPE_FORBIDDEN_END -->/{f=0} f && /^- `/{gsub(/^`|`$/,"",$2); print $2}' design.md
```

**设计理由**：
- HTML 注释是已有的框架惯例，无需引入新机制
- awk 区间解析可移植（不依赖 grep -P）
- 标记唯一，不会误读正文中的同名文本
- 每个 glob 有 `—` 后的人类可读说明

## 十、迁移策略

### 10.1 新项目

- `scripts/init-dossier.sh` 不再创建 work-items/ 目录
- install-workspace.sh 不再复制 work-item.yaml 模板
- meta.yaml 新模板不含 focus_work_item / active_work_items / execution_* 字段

### 10.2 已有项目升级

**不在 install-workspace.sh 中删除项目数据**（它不知道项目根目录，且旧 WI 可能含未沉淀信息）。

新增显式迁移命令 `codespec migrate-remove-wi <project_root>`：

1. **检查前提**：project_root 处于 Design 或更早阶段（未进入 Implementation）
2. **信息转换**：如果 work-items/ 存在，读取每个 WI yaml 的 allowed_paths / forbidden_paths，生成 design.md §4 实现边界的 glob 列表（需要人工确认）
3. **备份**：`mv work-items/ work-items.bak/`
4. **meta.yaml 清理**：删除 focus_work_item、active_work_items、execution_group、execution_branch、feature_branch 字段
5. **提示**：告知用户更新项目级 CLAUDE.md / AGENTS.md 中的 WI 引用

如果项目已处于 Implementation 或之后阶段：提示用户先完成当前回路或手动迁移，不自动操作。

### 10.3 已归档版本

`versions/*/work-items/` 保留不动。归档是只读快照，不需要迁移。

### 10.4 兼容性说明

升级后的框架不会主动读取 work-items/ 目录。如果 AI agent 或 review 文档引用旧 WI，它们只是无效引用，不影响流程。

### 10.5 已删除命令的友好提示

`add-work-item`、`set-active-work-items`、`set-execution-context` 不直接删除命令路由，改为返回友好错误：

```
ERROR: add-work-item 已移除。请直接编辑 design.md §7 实现计划。
```

这样迁移中的用户不会看到 confusing 的 "unknown command"。
